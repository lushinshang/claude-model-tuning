本導讀旨在拆解 Anthropic 官方發布之 《Choosing a Claude model and effort level in Claude Code》，協助開發者在日常人機協同中建立更精準的直覺：何時該升級模型（Model），何時該提高心力（Effort）？

💡
吳恩達（Andrew Ng）思維濾鏡
許多開發者誤以為 AI 是在對話中學習的，但事實上，模型在推理階段（Inference）的本質是「靜態預測」。理解這點，是建立自動化 Agent 工作流的起點。在 Agentic Workflow 中，『規劃』與『自我修正』是核心。心力控制（Effort Control）本質上是調整 Agent 在執行 Loop 中的『步幅與驗證閥值』。

第一幕：地圖與疆域的本質——唯讀權重與引導 (Weights & Steering)

當我們在 Claude Code 中按下 Enter 時，背後發生了什麼？

圖一：Claude API 推理與切詞工作流
                graph TD
                    A[使用者訊息 + CLAUDE.md + 對話歷史 + 上下文檔案] -->|打包發送| B[API 請求]
                    B -->|伺服器端 Tokenization| C[Token ID 整數陣列]
                    C -->|矩陣乘法| D[凍結的權重 Frozen Weights]
                    D -->|機率計算| E[預測下一個 Token]

1. 唯讀權重（Frozen Weights）的本質

當你的請求被送往 API 時，伺服器會先進行 Tokenization（切詞並映射為 vocabulary 的整數 ID）。接著，模型會將這些整數 ID 丟進矩陣，經過一連串的矩陣乘法，輸出下一個 token 的機率分佈。在這個過程中，模型權重是完全凍結的。

  • 你的 Prompt、你的 CLAUDE.md 或是你放入 Context 的上萬行程式碼，沒有任何一行會修改模型的權重。
  • Steering vs. Teaching：把文件放入 Context 只是在「引導（Steering）」機率分佈，並非真的在「教導（Teaching）」模型。這種影響僅限於該次請求，模型並不會因此變聰明或「記住」這項知識。

2. 幻覺的真相

當 Claude 很有自信地呼叫了一個根本不存在的 API（幻覺）時，這並不是因為它「內部資料庫查詢失敗」，而是因為唯讀權重根據訓練時留下的統計模式，計算出這個 token 序列「看起來最合理」。

地圖（Map）是模型唯讀權重中所保留的統計機率模型，而疆域（Territory）則是程式碼的真實執行環境。如果一個庫是在模型訓練截止日期後才發布的，那它絕對不會出現在「地圖」中，你只能透過 Context 來引導（Steer）它。

🎨
設計概念：地圖與疆域的落差 (Map vs Territory)
推薦使用日系可愛風格 (Taiwan kawaii) 繪製：左邊一隻 Chibi 工程師抱著寫有「地圖」的發光卷軸，右邊是佈滿晶片紋路與程式塊的島嶼「疆域」。兩者中間的鴻溝站著一朵困惑的雲朵小精靈「未知未知」。
📈
簡立峰思維濾鏡
在企業級導入 AI 時，最大的浪費來自於「用高昂的運算成本處理簡單問題」，或是「用低能力配置處理高複雜任務而導致專案失敗」。這是一場經典的運算經濟學遊戲。大模型的 per-token 價格極高，但如果用小模型反覆嘗試（Iterations）失敗，所累積的 tokens 費用與時間成本反而更高。開發者必須建立對應的 Heuristic 與 ROI 戰略。

第二幕：通才、專家與專科專家——三種模型心智 (Sonnet, Opus & Fable)

官方針對 Claude 家族在 Claude Code 中的角色，給出了一個非常生動的比喻。我們可以將它們視為三種不同層級的醫生：

角色層級 代表模型 特質與心智模型 Low Effort (低心力) 表現 High Effort (高心力) 表現
專科專家 (Specialist) Claude Fable 5 見過幾乎沒有人看過之疑難雜症的頂尖專家。收費最高，但能解決其餘模型無法觸及的深水區難題。 瞥一眼就能精準指出大家卡住的關鍵盲點。 會調動極大資源，針對極度複雜的多步驟重構進行全面推導。
專家 (Expert) Claude Opus 具備深厚工程直覺的資深工程師。擁有豐富的外部模式識別能力。 像諮詢專家 5 分鐘。能迅速指出架構 Gotchas，但不會仔細查閱所有程式碼細節。 仔細通讀程式碼，並結合其深厚直覺，給出面面俱到的優化方案。
優秀通才 (Generalist) Claude Sonnet 執行力極強、博聞強記的全科醫生。對上下文中的特定程式碼理解極深。 能快速執行精準定義的常規修改，速度快且成本極低。 像給通才整個下午。它會讀完所有相關檔案、跑測試、反覆檢查，對你的 codebase 了若指掌，但缺少「一眼看出程式碼外架構盲點」的經驗直覺。

總結來說：Model 代表「多有能力」（How Capable），決定了天花板高度與固化知識廣度；而 Effort 代表「多徹底」（How Thorough），決定了它在此天花板之下願意投入多少資源進行探索與驗證。

第三幕：心力 (Effort) 到底在算什麼？ (The Engine of Execution)

在 Claude Code 中，心力是一個控制 Claude 實際工作量的手動旋鈕，它影響的是整體的執行引擎：

  • 讀取更多檔案：主動探索更廣泛的關聯代碼。
  • 增加驗證步驟：更頻繁地跑測試、跑靜態分析或編譯器。
  • 多步驟推進（Step Limit）：在暫停下來向你詢問/確認之前，它會自主嘗試推得更遠、解決更多中途遇到的依賴問題。

不論是 Claude 的推理思維鏈（Thinking）、調用工具（Tool calls），還是最後對你說的話（Text to you），在 API 底層都是同一種輸出 Token，以同樣的費率計費。在高心力模式下，Claude 會在第一步先建立計畫。但這個計畫是動態的：如果計畫有 3 個除錯假說，在執行第 1 步時就跑測試抓到了 bug，Claude 會動態修剪計畫，跳過假說 2 與 3，直接進入修復與合併，避免浪費額外 Token。

🤝
陳縕儂思維濾鏡
人機互動的關鍵在於「期望管理」與「主導權的合理分配」。開發者不能把 AI 當作黑盒子，必須隨時理解自己是在與哪一種特質的模型對話。此外，設計良好的人機對齊循環（如實作後的隨堂測驗）能讓開發者更安心，也更容易建立起互信的開發默契。

第四幕：黃金診斷法則 (Heuristics for Debugging)

當 Claude Code 給出的答案錯了，開發者不應該盲目亂調旋鈕，而是應該依循以下診斷路徑：

圖二：Claude Code 報錯後的診斷路徑
                flowchart TD
                    A[Claude Code 給出錯誤答案] --> B{檢查 Context 上下文}
                    B -->|Context 不足或模糊| C[優化 Prompt / 修改 CLAUDE.md / 精確指定檔案]
                    B -->|Context 充足且 Claude 已經盡力| D{分析錯誤原因}
                    
                    D -->|它真的不懂 / 能力不足 / 邏輯死胡同| E[調整 Model: 換用更大模型]
                    D -->|它偷懶 / 漏看檔案 / 沒跑測試 / 寫一半放棄| F[調整 Effort: 提高心力等級]

1. 調整 Model 的訊號：問題「太難了」

特徵：小模型（Sonnet）給出了非常自信的錯誤答案。你給了它完整的 Context，甚至一行行教它，它依然陷入邏輯死胡同。此時你應該升級模型（例如從 Sonnet 升級至 Opus 或 Fable）。

2. 調整 Effort 的訊號：Claude「不夠努力」

特徵：Claude 原本可以寫對，但它為了省 Token，選擇不去讀某個定義檔，或者修改完代碼後沒有跑測試，導致漏掉邊際狀況；或者重構到一半直接放棄向你提問。此時你應該提高 Effort，迫使它進行更徹底的雙重檢查。

第五幕：邊際成本與 Token 經濟學 (Token Economics)

如何取得 ROI 的最佳平衡點？這是一個動態的過程：

  • 例行工作：使用小模型（Sonnet）+ 預設/低 Effort。大模型在這種任務上會產生多餘的驗證步驟，造成雙重浪費。
  • 複雜任務：使用大模型（Fable / Opus）+ 高 Effort。小模型在極限邊緣掙扎時，會不斷出錯、重試，燒掉大量的 iteration loops，累積龐大的 token 帳單；大模型通常能一次到位，最終的總成本反而更低。
  • 控制邊界：max_tokens 會像一堵牆一樣直接在中間粗暴截斷 output，不建議作為日常控制手段。相反地，在 Prompt 中加上「keep it brief（保持簡潔）」或設定 task budget 是更好的軟性引導。

🛠️ 開發者工具箱：黃金對齊 Prompts

以下工具卡片提供了一鍵複製功能，建議在開發時貼入對話視窗使用：

1. 實作前：盲點檢視 PRE-IMPLEMENTATION
我們即將開始實作以下任務：[填入任務描述]。 在開始寫任何程式碼或制定計畫之前，請先進行「盲點檢視（Blind Spot Pass）」： 1. 列出你目前 Context 中缺失的關鍵上下文（例如未讀取的檔案、環境配置、依賴庫版本）。 2. 指出該任務中可能存在的「未知未知（Unknown Unknowns）」，並提出 3 個你需要我澄清的關鍵設計決策。 先不要提供實作計畫，等我回覆你的提問。
2. 實作中：偏差記錄 DURING-IMPLEMENTATION
在接下來的修改中，如果發現實際程式碼現況與我們原先的設計計畫有落差，或是需要做出折衷設計（Trade-off），請立即暫停並更新我們的 `implementation-notes.md`： 1. 記錄偏離的原因與當前的程式碼限制。 2. 列出當前決策對系統其他模組的潛在影響（Side Effects）。 3. 提出修正後的替代方案，等我確認後再繼續。
3. 實作後：隨堂測驗 POST-IMPLEMENTATION
你已經完成了代碼修改。在我們進行 Git Commit 與 Merge 之前，請為我出一份包含 3 個問題的「隨堂測驗（Quiz）」： 1. 針對你剛剛修改中最微妙的邏輯、臨界值處理（Edge Case）或潛在的 Concurrency 點進行提問。 2. 題目必須是選擇題或簡答題，旨在測試我是否真正理解你的修改，而非盲目相信你的代碼。 請只出題，等我回答後再批改並給出解釋。
